# PinLog 데이터 모델 및 무결성

## 1. 공통 원칙

- 사용자 소유 데이터는 `deleted_at` 기반 소프트 삭제를 사용합니다.
- 사용자 조회에서는 활성 데이터만 반환합니다.
- Place는 공용 데이터이므로 User 데이터 삭제와 함께 삭제하지 않습니다.
- 시간 필드는 `created_at`, `updated_at`, `deleted_at` 명칭을 사용합니다.
- 공식 엔티티명은 공식 용어사전을 따릅니다.

## 2. 주요 테이블

```text

updated_at
deleted_at
```


### records

```text

place_id
status

created_at
updated_at
deleted_at
```

활성 상태의 동일 User–Place Record는 하나만 허용합니다.

```sql
CREATE UNIQUE INDEX uq_active_record_user_place
ON records(user_id, place_id)
WHERE deleted_at IS NULL;
```


## 3. 정합성 규칙

| 규칙 | 보장 방식 |
|---|---|
| User당 활성 Record는 Place별 최대 1개 | 부분 유니크 인덱스 |
| Record는 Context 1개 이상 | 생성·삭제 서비스 트랜잭션 |
| Collection은 Record 1개 이상 | 생성·제거 서비스 트랜잭션 |
| Collection 내 동일 Record 중복 금지 | 활성 연결 유니크 제약 |
| User당 Shelf 1개 | `UNIQUE(user_id)` |
| 동일 Shelf 중복 Follow 금지 | 활성 Follow 유니크 제약 |
| 자기 Shelf Follow 금지 | 애플리케이션 또는 DB 트리거 |
| 삭제 데이터 미노출 | 모든 조회의 활성 데이터 필터 |

## 4. 핵심 트랜잭션

### Record 생성

```text
카카오 Place 조회 또는 저장
→ Record 생성
→ 첫 Context 생성
→ Keyword 관계 생성
→ 임베딩 생성
→ Record 활성화
```

### Record 재생성

```text
기존 Record 잠금
→ Collection 연결 조회
→ 기존 CollectionRecord 소프트 삭제
→ 기존 Record 소프트 삭제
→ 새 Record 생성
→ 활성 Context 연결
→ CollectionRecord 재생성
→ Keyword 관계 재생성
→ 기존 임베딩 검색 제외
→ 전체 활성 Context 재임베딩
→ 새 Record 활성화
```

### 마지막 Context 삭제

```text
마지막 Context 여부 확인
→ 사용자 확인
→ Collection 영향 확인
→ Context·Record·필요 Collection 소프트 삭제
```

### 회원 탈퇴

```text
User 관련 데이터 소프트 삭제
→ User가 생성한 Follow 소프트 삭제
→ User Shelf 대상 Follow 소프트 삭제
→ Feed·Library 노출 제외
```

## 5. 구현 주의점

- Context 행을 새 Record로 이동하는 방식은 과거 Record 시점의 Context 스냅샷을 보존하지 않습니다.
- Record 재생성으로 `created_at`이 바뀌면 Collection 내 정렬 위치도 바뀝니다.
- Feed와 Library 조회는 User, Shelf, Collection, Record, CollectionRecord의 삭제 상태를 모두 확인해야 합니다.
- 새 Record는 Context 연결, Collection 연결, Keyword 관계, 임베딩이 완료되기 전에 활성 조회 대상이 되어서는 안 됩니다.
