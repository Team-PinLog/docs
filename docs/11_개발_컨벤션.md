이 문서는 기존 Release 문서와 기술 스택에서 확인된 규칙만 정리합니다. 별도 합의가 확인되지 않은 브랜치명, 커밋 형식, 테스트 커버리지, 포매터 규칙은 정의하지 않습니다.

## 2. 용어와 이름

- 코드, API, DB, UI, 문서에서 공식 용어를 사용합니다.
- `Memory`, `장소 모음집`, `익명 책장`, `발견 피드`, `개인 Archive`는 사용하지 않습니다.
- 엔티티명은 `User`, `SocialAccount`, `Place`, `Record`, `Context`, `Keyword`, `Collection`, `CollectionRecord`, `Shelf`, `Follow`, `Embedding`을 기준으로 합니다.
- DB 테이블과 컬럼은 snake_case를 사용합니다.
- 시간 컬럼은 `created_at`, `updated_at`, `deleted_at`을 사용합니다.
- REST Endpoint는 복수 명사와 소문자 경로를 사용합니다.

## 3. 데이터 접근

- 사용자 소유 데이터 삭제는 물리 삭제가 아니라 소프트 삭제입니다.
- Place는 공용 데이터이므로 User 데이터 삭제와 함께 삭제하지 않습니다.
- 사용자 조회에는 활성 데이터 필터를 적용합니다.
- Feed와 Library에서는 User, Shelf, Collection, Record, CollectionRecord의 삭제 상태를 모두 검사합니다.
- 공개 조회에서 Context 원문과 사용자 신원 정보를 반환하지 않습니다.

## 4. 무결성

- 활성 `Record`는 User–Place 조합당 최대 하나입니다.
- Record 생성 시 첫 Context를 함께 생성합니다.
- 빈 Collection은 생성하지 않습니다.
- Collection 안에 같은 Record를 중복 연결하지 않습니다.
- User당 Shelf는 하나입니다.
- 자기 Shelf Follow와 동일 Shelf 중복 Follow를 허용하지 않습니다.
- Follow 별칭은 앞뒤 공백 제거 후 빈 값이면 `null`로 저장합니다.

## 5. 트랜잭션

다음 작업은 서비스 트랜잭션 경계에서 처리합니다.

- Place 조회·저장과 Record·첫 Context 생성
- Record 재생성과 Collection 연결 승계
- 마지막 Context 삭제와 Record·Collection 연쇄 처리
- 마지막 CollectionRecord 제거와 Collection 삭제
- 회원 탈퇴와 관련 데이터·Follow 소프트 삭제

Record 재생성 완료 전에는 새 Record를 활성 조회 대상으로 노출하지 않습니다.

## 6. AI 처리

- 기능명은 `AI 자연어 검색`을 사용합니다.
- 검색 대상은 로그인 User의 Place, Context, Keyword입니다.
- 검색 결과는 Record 단위로 중복 제거합니다.
- Context 생성·수정 시 관련 임베딩을 생성 또는 재생성합니다.
- Record 재생성 시 모든 활성 Context와 Keyword 검색 표현을 새 Record 기준으로 재생성합니다.

## 7. 정렬과 공개

- Collection과 Collection 내부 Record는 정책상 Record `created_at` 오름차순을 따릅니다.
- Collection 생성은 자동 발행을 포함합니다.
- 공개 Collection 수정은 별도 발행 스냅샷 없이 현재 공개본에 반영합니다.
