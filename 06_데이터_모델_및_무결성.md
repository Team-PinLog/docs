## 2. Record 재생성

Record 수정 시 다음 작업을 하나의 서비스 트랜잭션으로 수행한다.

```text
기존 Record 잠금
→ 연결 Collection 조회
→ 기존 CollectionRecord 소프트 삭제
→ 기존 Record 소프트 삭제
→ 새 Record 생성(previous_record_id 설정)
→ Context의 record_id를 새 Record로 연결
→ CollectionRecord 새 행 생성
→ Keyword 관계 재생성
→ 기존 임베딩 소프트 삭제
→ 전체 활성 Context 재임베딩
```

## 3. 중요한 구현 주의점

### Context 이력

현재 정책은 `Record만 새 생성`이다. 따라서 Context 행을 그대로 새 Record로 이동하면 과거 Record의 완전한 스냅샷은 남지 않는다.

즉, 보존되는 것은:

- 과거 Record 행
- 재생성 연결 관계
- 현재 Context

보존되지 않는 것은:

- 과거 Record 시점의 Context 구성 스냅샷

완전한 이력이 필요해지면 향후 Context 버전 테이블 또는 Context 복제 정책이 필요하다.

### Collection 정렬

정책상 Record `created_at ASC`를 사용한다.

Record 재생성 시 생성일이 새로 설정되므로 수정된 Record는 Collection 뒤쪽으로 이동한다.

이는 정책에 따른 의도된 동작으로 본다.

### 소프트 삭제 필터

모든 사용자 조회 쿼리에 `deleted_at IS NULL`을 적용한다.

특히 Feed와 Library에서는 User, Shelf, Collection, Record, CollectionRecord를 모두 확인해야 한다.

## 4. 임베딩 무결성

Record가 ACTIVE 상태가 되기 전에 다음이 완료되어야 한다.

- Context 연결
- Collection 연결 승계
- Keyword 관계
- 전체 활성 Context 임베딩
- Record 검색 문서 생성

실패 시 새 Record를 사용자 검색과 Feed에서 제외한다.
