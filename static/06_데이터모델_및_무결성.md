# PinLog 데이터 모델 및 무결성

## 1. 공통 원칙

### 1.1 삭제 정책

- 사용자 소유 데이터는 `deleted_at` 기반 소프트 삭제를 사용합니다.
- **삭제 데이터는 복구하지 않습니다.** 삭제 후 재저장은 기존 행의 재활성화가 아니라 항상 새 INSERT입니다.
- 따라서 모든 유니크 제약은 **활성행(`deleted_at IS NULL`)에만 적용하는 부분 유니크 인덱스**로 정의합니다. 전체 유니크로 정의하면 삭제 후 재저장이 실패합니다.
- 사용자 조회에서는 활성 데이터만 반환합니다.
- Place는 공용 데이터이므로 삭제하지 않습니다. `deleted_at` 컬럼을 두지 않습니다.
- AI 파생 데이터(임베딩·키워드)는 소프트 삭제하지 않고 **즉시 파기**합니다.

### 1.2 기술 전제

- DBMS는 PostgreSQL입니다. 부분 유니크 인덱스와 `CHECK` 제약을 적극 사용합니다.
- 벡터 검색은 pgvector 확장을 사용하며, 벡터는 동일 인스턴스 내 `ai` 스키마에 있습니다.
- 시간 필드는 `created_at`, `updated_at`, `deleted_at` 명칭과 `TIMESTAMPTZ` 타입을 사용합니다.
- 기본키는 서로게이트 키(`BIGINT GENERATED ALWAYS AS IDENTITY`)를 사용합니다. 외부 시스템 식별자(`kakao_place_id`)를 기본키로 사용하지 않습니다.

### 1.3 스키마 소유

| 스키마 | 소유 | 테이블 |
|---|---|---|
| `core` | 백엔드 | member, social_account, place, record, context, collection, collection_record, follow |
| `ai` | AI 파트 | keyword_preset, context_embedding, context_keyword |

동일 PostgreSQL 인스턴스 내 스키마 분리이므로 크로스 스키마 조인이 가능합니다.

쓰기 매트릭스:

| 테이블 | 백엔드 | AI 워커 |
|---|---|---|
| `core.*` | O | X |
| `ai.context_embedding` | 삭제 시 DELETE만 | INSERT / UPDATE |
| `ai.context_keyword` | 삭제 시 DELETE만 | INSERT / DELETE |
| `ai.keyword_preset` | X | O |

백엔드는 `ai.context_keyword`를 **읽기 조인**할 권리를 가집니다. Record·Collection·Feed 조회에서 공개 Keyword를 제공해야 하기 때문입니다.

### 1.4 도메인 용어와 물리 테이블

공식 용어사전의 도메인 용어와 물리 테이블명이 일부 다릅니다.

| 도메인 용어 | 물리 테이블 | 비고 |
|---|---|---|
| User | `member` | `user`는 PostgreSQL 예약어이므로 `member` 사용 |
| Shelf | **없음** | 조회 결과의 그룹핑 단위이며 물리 테이블이 아님 |
| Library | **없음** | 내 Collection + 팔로우한 Shelf의 Collection을 모은 조회 결과 |

Shelf는 User와 1:1이면서 고유 속성이 없습니다. 구분 기준은 `collection.member_id`, 표시 이름은 `follow.display_name`, 내용물은 Collection 목록으로 모두 다른 테이블에 존재합니다. 따라서 테이블로 두지 않습니다.

```sql
-- 내 선반
SELECT * FROM collection
WHERE member_id = :me AND deleted_at IS NULL;

-- 라이브러리(팔로우한 선반들)
SELECT f.followee_member_id AS shelf_key, f.display_name, c.*
FROM follow f
JOIN member m     ON m.id = f.followee_member_id AND m.deleted_at IS NULL
JOIN collection c ON c.member_id = f.followee_member_id AND c.deleted_at IS NULL
WHERE f.follower_member_id = :me AND f.deleted_at IS NULL
ORDER BY f.created_at DESC, c.created_at DESC;
```

`shelf_key`로 그룹핑한 결과가 Shelf이고, 그 묶음이 Library입니다.

## 2. 테이블 정의

### 2.1 member

```text
id          BIGINT PK
created_at  TIMESTAMPTZ NOT NULL
deleted_at  TIMESTAMPTZ NULL
```

- 신원과 인증 수단을 분리합니다(정책 2.2). 인증 정보는 `social_account`가 소유합니다.
- `id`는 내부 식별자이며 **API 응답에 노출하지 않습니다**. 5장을 참고합니다.
- 익명 서비스이므로 저장하는 개인정보가 없습니다. 컬럼이 적은 것이 정상입니다.
- 삭제 판정 기준은 `deleted_at` 하나입니다. `status` 컬럼을 두지 않습니다.

### 2.2 social_account

```text
id                BIGINT PK
member_id         BIGINT FK -> member
provider          VARCHAR(20)   -- GOOGLE / KAKAO / NAVER
provider_user_id  VARCHAR(255)
email             VARCHAR(255) NULL
created_at        TIMESTAMPTZ NOT NULL
deleted_at        TIMESTAMPTZ NULL
```

- `provider_user_id`는 공급자가 발급한 식별자입니다(Google `sub`, Kakao `id`, Naver `response.id`). 식별 기준으로는 이메일이나 닉네임을 사용하지 않습니다.
- 공급자 간 값이 충돌할 수 있으므로 유니크는 `(provider, provider_user_id)` 복합입니다.
- 숫자로 보이는 값도 문자열로 저장합니다.
- `email`은 설정 화면 표시용으로 저장합니다. 공급자가 제공하며, 미동의·미제공 시 `null`일 수 있습니다. 식별키가 아니므로 유니크를 걸지 않습니다.
- 탈퇴 시 `provider_user_id`와 `email`을 마스킹합니다. 개인정보이므로 파기 대상입니다.
- `member`와 별도로 `deleted_at`을 둡니다. 부분 유니크 인덱스의 `WHERE` 절이 자기 테이블 컬럼만 참조할 수 있기 때문이며, 이 컬럼이 없으면 탈퇴 후 동일 소셜 계정으로 재가입할 수 없습니다.

### 2.3 place

```text
id              BIGINT PK
kakao_place_id  VARCHAR(50)  NOT NULL
name            VARCHAR(100) NOT NULL
address         VARCHAR(200) NOT NULL   -- 지번 주소(address_name), 동 포함
road_address    VARCHAR(200) NULL       -- 도로명 주소(road_address_name)
phone           VARCHAR(30)  NULL
place_url       VARCHAR(300) NULL
lat             DECIMAL(10,7) NOT NULL
lng             DECIMAL(10,7) NOT NULL
created_at      TIMESTAMPTZ NOT NULL
updated_at      TIMESTAMPTZ NOT NULL
```

**식별**

- 식별 기준은 `kakao_place_id`입니다. 좌표는 표시·지도용이며 식별키가 아닙니다.
- 좌표를 식별키로 쓰지 않는 이유: 카카오가 동일 장소의 좌표를 미세 조정하고, 한 좌표에 다수 점포가 존재하며, 층·호 정보가 카카오 로컬 API 응답에 없습니다. 좌표 기반 매칭은 확률적이라 병합 오류가 누적됩니다.

**카카오 로컬 API 매핑**

| 컬럼 | 카카오 응답 필드 |
|---|---|
| `kakao_place_id` | `id` (문자열) |
| `name` | `place_name` |
| `address` | `address_name` (지번, 동 포함) |
| `road_address` | `road_address_name` (없을 수 있음) |
| `phone` | `phone` |
| `place_url` | `place_url` |
| `lat` | `y` (문자열 → 파싱) |
| `lng` | `x` (문자열 → 파싱) |

- 좌표는 문자열로 응답되므로 파싱이 필요하며, `x`가 경도(lng), `y`가 위도(lat)입니다.
- `address`(지번)와 `road_address`(도로명)를 카카오가 주는 대로 모두 저장합니다. 표시는 도로명 우선, 없으면 지번을 사용합니다.
- **지역(area) 라벨**은 별도 컬럼을 두지 않고 `address`(지번)의 동(洞) 부분에서 파생합니다. 컬렉션 목차 그룹핑 등 표시용이며, 지역 검색은 자연어 검색이 커버하므로 지역 전용 검색·컬럼을 두지 않습니다.
- `distance`는 요청 시 넘긴 중심 좌표 기준 상대값이므로 저장하지 않고 응답 DTO에서만 사용합니다.
- `category_group_code`는 값이 비어 있는 경우가 있고 추천이 Keyword 기반이므로 저장하지 않습니다.

**저장 시점과 갱신**

- 검색 결과를 조회하는 것만으로 저장하지 않습니다. Record 생성 시점에만 저장합니다(정책 4.3).
- 장소 검색은 **프론트가 카카오 로컬 API를 직접 호출**하며, Record 생성 시 카카오 응답의 장소 데이터를 서버에 전달합니다. 서버는 `kakao_place_id` 기준으로 upsert하고, 서버가 카카오를 호출하지 않습니다.
- 저장 후 갱신하지 않습니다. Place 행은 **저장 시점 스냅샷**으로 취급하며(동일 `kakao_place_id` 재전달 시 기존 행 재사용, 갱신 없음), 최신 정보는 `place_url`을 통해 카카오맵에 위임합니다.
- 조회(모아보기·지도)는 DB 전용입니다. 조회 시 카카오를 재호출하지 않습니다.

### 2.4 record

```text
id          BIGINT PK
member_id   BIGINT FK -> member
place_id    BIGINT FK -> place
created_at  TIMESTAMPTZ NOT NULL
updated_at  TIMESTAMPTZ NOT NULL
deleted_at  TIMESTAMPTZ NULL
```

- Record는 한 User와 한 Place의 연결 단위이며, 활성 Context를 최소 한 개 가져야 합니다.
- `status`와 `previous_record_id`를 두지 않습니다. 삭제 판정은 `deleted_at` 하나이고, 삭제된 Record와 새 Record를 연결할 필요가 없습니다.
- 동일 장소에 여러 저장 이유를 남기려면 새 Record가 아니라 기존 Record에 Context를 추가합니다.

### 2.5 context

```text
id          BIGINT PK
record_id   BIGINT FK -> record
member_id   BIGINT FK -> member   -- 비정규화
body        TEXT NOT NULL
created_at  TIMESTAMPTZ NOT NULL
deleted_at  TIMESTAMPTZ NULL
```

- `body`는 개인정보입니다. 타인에게 공개하지 않습니다(정책 5.7).
- `member_id`는 비정규화 컬럼입니다. 자연어 검색이 항상 본인 맥락으로 한정되므로(정책 13.1) Record 조인 없이 `WHERE member_id = ? AND deleted_at IS NULL`로 후보를 좁힌 뒤 벡터 연산을 수행하기 위해 둡니다.
- **Context는 불변입니다.** 사용자에게 제공되는 조작은 추가와 삭제뿐이며, 수정은 기존 Context를 삭제하고 새 Context를 생성하는 것으로 표현합니다.
- 불변이므로 `updated_at`과 버전 컬럼을 두지 않습니다.
  - 버전 컬럼은 비동기 결과가 낡았는지 판별하기 위한 장치였습니다. 본문이 바뀌지 않으므로 판별 대상 자체가 없습니다.
  - 서로 다른 Context에 대한 비동기 작업은 각각 다른 행에 기록되므로, 완료 순서가 뒤바뀌어도 덮어쓰기가 발생하지 않습니다.
- **수정 시 `created_at`은 새 행의 생성 시각으로 갱신됩니다(M2 A안 결정, 백엔드 주도).** 수정은 삭제+새 행 생성이므로, 목록 정렬·날짜 표시에서 수정된 맥락이 최신으로 취급되는 것은 **의도된 동작**입니다(버그가 아닙니다). 최초 작성 시각을 보존하거나 계보를 추적해야 할 요구가 생기면 `origin_context_id` 도입을 재검토합니다.
- 임베딩과 Keyword는 `ai` 스키마가 소유합니다. `context` 테이블에 임베딩 컬럼을 두지 않습니다.

### 2.6 collection

```text
id            BIGINT PK
member_id     BIGINT FK -> member
title         VARCHAR(20) NOT NULL
is_published  BOOLEAN NOT NULL DEFAULT TRUE
published_at  TIMESTAMPTZ NULL
record_count  INT NOT NULL DEFAULT 0
created_at    TIMESTAMPTZ NOT NULL
updated_at    TIMESTAMPTZ NOT NULL
deleted_at    TIMESTAMPTZ NULL
```

- 소유자는 `member_id` 하나입니다. `shelf_id`를 두지 않습니다.
- 활성 Collection에는 활성 CollectionRecord가 최소 한 개 있어야 합니다.
- `is_published`는 MVP에서 항상 `true`로 생성되지만, 향후 공개·비공개 전환을 위해 데이터로 관리합니다(정책 7.2).
- `record_count`는 활성 연결 수의 비정규화 값입니다. 목록·피드에서 장소 개수(정책 8.1)를 N+1 없이 제공하기 위해 두며, 연결 추가·제거와 **동일 트랜잭션**에서 갱신합니다.

### 2.7 collection_record

```text
id             BIGINT PK
collection_id  BIGINT FK -> collection
record_id      BIGINT FK -> record
created_at     TIMESTAMPTZ NOT NULL
deleted_at     TIMESTAMPTZ NULL
```

- Record는 Collection의 하위 데이터가 아닙니다. Collection에 속하지 않은 Record가 존재할 수 있고, 하나의 Record가 여러 Collection에 포함될 수 있습니다.
- 컬렉션 내부 정렬 기준은 `collection_record.created_at DESC`(담은 순서 최신순)입니다. Record의 `updated_at`을 기준으로 삼으면 Context 수정만으로 순서가 바뀌므로 사용하지 않습니다.
- `record_id` 단독 인덱스가 필요합니다. Record 삭제 시 "해당 Record가 마지막인 Collection"을 역조회하는 데 사용합니다(정책 6.8).

### 2.8 follow

```text
id                  BIGINT PK
followee_member_id  BIGINT FK -> member
follower_member_id  BIGINT FK -> member
display_name        VARCHAR(20) NULL
created_at          TIMESTAMPTZ NOT NULL
deleted_at          TIMESTAMPTZ NULL
```

- Shelf 테이블이 없으므로 팔로우 대상을 `member`로 직접 참조합니다. 이 덕분에 자기 팔로우 금지를 DB `CHECK`로 보장할 수 있습니다.
- `display_name`은 Shelf가 아니라 **팔로우 관계에 속합니다**. 같은 Shelf도 팔로워마다 다른 이름을 가질 수 있습니다.

| 항목 | 정책 |
|---|---|
| 이름 입력 | 선택 |
| 미지정·공백만 입력 | `null`로 저장 |
| 최대 길이 | 20자 |
| 앞뒤 공백 | 제거 후 저장 |
| 중복 이름 | 허용 |
| 수정·제거 | 허용 |

### 2.9 ai 스키마 (참조)

AI 파트 소유이며 상세 정의는 AI 설계 문서를 따릅니다.

```text
ai.keyword_preset
    id, code, label, description, examples,
    embedding, visibility, version, embedding_profile, active

ai.context_embedding
    id, context_id, embedding, model, status
    -- UNIQUE(context_id), is_deleted 없음(즉시 파기), HNSW 인덱스

ai.context_keyword
    context_id, keyword_id, confidence, preset_version
```

- **파이프라인 처리 단위는 Context입니다.** Context가 추가되면 그 Context만 임베딩·Keyword를 생성하고, 삭제되면 그 Context의 파생 데이터만 파기합니다.
- Keyword는 **Context 단위**로 저장합니다. Record 단위로 저장하면 Context를 삭제할 때 어느 Keyword를 제거할지 판별할 수 없어 Record 전체 재분석이 필요해집니다.
- `ai.context_embedding`은 `UNIQUE(context_id)`를 둡니다. Context 하나당 임베딩 하나로 고정하면 재시도 시 중복 INSERT가 원천 차단됩니다.
- 버전 컬럼을 두지 않습니다. Context가 불변이고 처리 단위가 Context이므로, 서로 다른 작업이 같은 행을 덮어쓰는 상황이 발생하지 않습니다.
- Record Keyword와 Collection Keyword는 저장하지 않고 조인 집계로 파생합니다. 피드 성능이 문제되면 그때 읽기 전용 캐시를 추가합니다.
- Keyword는 사전 정의된 프리셋에서만 선택합니다. 이 제약이 공개 Keyword의 안전성을 보장하는 장치입니다.

## 3. 제약과 인덱스

### 3.1 유니크 제약

모두 활성행 기준 부분 유니크 인덱스입니다. `place`만 전체 유니크입니다.

```sql
CREATE UNIQUE INDEX uq_place_kakao
    ON place (kakao_place_id);

CREATE UNIQUE INDEX uq_social_active
    ON social_account (provider, provider_user_id)   WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX uq_record_active
    ON record (member_id, place_id)                  WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX uq_colrec_active
    ON collection_record (collection_id, record_id)  WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX uq_follow_active
    ON follow (followee_member_id, follower_member_id) WHERE deleted_at IS NULL;
```

### 3.2 CHECK 제약

```sql
ALTER TABLE follow ADD CONSTRAINT ck_follow_self
    CHECK (followee_member_id <> follower_member_id);

ALTER TABLE context ADD CONSTRAINT ck_context_body
    CHECK (length(btrim(body)) > 0);

ALTER TABLE collection ADD CONSTRAINT ck_collection_title
    CHECK (length(btrim(title)) > 0);

ALTER TABLE collection ADD CONSTRAINT ck_collection_count
    CHECK (record_count >= 0);

ALTER TABLE place ADD CONSTRAINT ck_place_lat CHECK (lat BETWEEN  -90 AND  90);
ALTER TABLE place ADD CONSTRAINT ck_place_lng CHECK (lng BETWEEN -180 AND 180);
```

좌표 범위 제약은 카카오 응답이 문자열이므로 파싱 오류를 잡아줍니다.

### 3.3 조회 인덱스

| 인덱스 | 용도 |
|---|---|
| `record (member_id, deleted_at)` | 지도·전체 Record 조회 |
| `record (place_id)` | Place 기준 조회 |
| `context (record_id, deleted_at)` | Record 상세의 Context 조회 |
| `context (member_id, deleted_at)` | 자연어 검색 범위 필터 |
| `collection_record (collection_id, created_at DESC)` | 컬렉션 내부 정렬 |
| `collection_record (record_id)` | Record 삭제 시 역조회 (필수) |
| `collection (member_id, created_at DESC)` | Shelf 조회 |
| `collection (is_published, published_at DESC) WHERE deleted_at IS NULL` | Feed 후보 스캔 |
| `follow (follower_member_id, created_at DESC)` | Library 조회 |
| `follow (followee_member_id)` | 팔로워 수 집계 |
| `place (lat, lng)` | 지도 bbox 조회 |
| `social_account (member_id)` | 계정 조회 |

팔로워 수와 팔로잉 수는 비정규화 컬럼을 두지 않습니다. 본인만 조회하므로(정책 3.3) 트래픽이 낮고, 컬럼을 두면 팔로우마다 쓰기 경합이 발생합니다.

## 4. 정합성 규칙

### 4.1 DB가 보장하는 것

| 규칙 | 보장 방식 |
|---|---|
| 동일 소셜 계정 중복 가입 금지 | 활성행 부분 유니크 |
| 동일 Kakao Place 중복 저장 금지 | `UNIQUE(kakao_place_id)` |
| User당 활성 Record는 Place별 최대 1개 | 활성행 부분 유니크 |
| Collection 내 동일 Record 중복 금지 | 활성행 부분 유니크 |
| 동일 Shelf 중복 Follow 금지 | 활성행 부분 유니크 |
| 자기 Shelf Follow 금지 | `CHECK` |
| Collection 제목 20자 이하·공백만 금지 | 컬럼 길이 + `CHECK` |
| Context 본문 공백만 금지 | `CHECK` |
| 좌표 범위 | `CHECK` |

### 4.2 애플리케이션이 보장하는 것

| 규칙 | 보장 방식 |
|---|---|
| 활성 Record는 활성 Context 1개 이상 | Context 삭제 시 Record 행 잠금 후 활성 수 확인. 1개면 409 거절 후 강제 삭제 API로만 처리 |
| 활성 Collection은 활성 Record 1개 이상 | 연결 제거 시 Collection 행 잠금 후 활성 수 확인. 1개면 409 거절 후 Collection 삭제 API로만 처리 |
| Context 수정 중 빈 Record 방지 | 추가를 먼저 하고 삭제를 나중에 수행 |
| `record_count` 정확성 | 연결 추가·제거와 동일 트랜잭션 |
| Record 삭제 시 마지막인 Collection 자동 삭제 | 역조회 후 동일 트랜잭션 처리 |
| 탈퇴 시 관련 데이터 정리 | 탈퇴 트랜잭션 또는 이벤트 기반 배치 |
| 삭제 데이터 미노출 | 모든 조회의 `deleted_at IS NULL` 필터 |
| 타인에게 Context 미노출 | 조회 쿼리·응답 DTO 분리 (5장) |

마지막 두 항목은 누락 시 즉시 사고로 이어지므로 구현 규약으로 관리합니다. Hibernate 6.4 이상이면 `@SQLRestriction("deleted_at IS NULL")`을 엔티티에 일괄 적용할 수 있습니다. 다만 삭제 행을 포함해야 하는 관리성 쿼리는 네이티브로 분리해야 합니다.

## 5. 공개 범위 구현 규약

### 5.1 공개 대상

| 리소스 | 소유자 | 타인 |
|---|---|---|
| Place | O | O |
| Record 생성 시각 | O | O |
| Keyword | O | O |
| **Context 본문** | O | **X** |
| Collection 제목·장소 수·생성일 | O | O (발행분) |

### 5.2 구현 규약

응답 조립 단계에서 필드를 걸러내는 방식은 누락 시 곧바로 개인정보 노출이 됩니다. 실패가 안전한 방향으로 나도록 **조회 단계에서 분리**합니다.

| 방식 | 실수했을 때 |
|---|---|
| 단일 쿼리 + DTO 필터링 | 개인정보 유출 |
| 공개용 복제 테이블 | 동기화 누락 시 낡은 데이터 노출 |
| **쿼리·DTO 분리 (채택)** | 필드가 없어 컴파일 오류 또는 null |

- **저장은 한 벌**로 유지합니다. 공개용 복제본을 만들지 않습니다. 이미 `ai.context_keyword`가 마스킹된 파생본 역할을 합니다.
- **쿼리 분리**: 타인 조회용 쿼리는 `context.body`를 조회하지 않습니다. 리포지토리 메서드명에 `ForOwner` / `Public`을 명시합니다.
- **DTO 분리**: 소유자용·공개용 응답 클래스를 상속 없이 별개로 정의합니다. 상속이나 조건부 직렬화는 부모에 필드가 추가될 때 조용히 노출됩니다.
- **경로 통합**: 타인이 데이터에 접근하는 경로는 Feed, Collection 상세, 팔로우한 Shelf 셋뿐입니다. 이 셋이 단일 공개 조회 서비스를 거치게 하여 노출 필드를 한 곳에서 관리합니다.

### 5.3 식별자 은닉

- `member.id`는 **공개 응답(타인 조회)에 포함하지 않습니다.** 순차 값이므로 노출 시 사용자 열거가 가능해지고, 가입 순번이 신원 추론 단서가 됩니다. 예외적으로 로그인·가입 확정 응답에만 본인의 id(`memberId`)를 반환합니다.
- 별도 공개 식별자(`public_id`)는 두지 않습니다. 대신 **Collection id를 진입점**으로 사용합니다.

```text
POST /follows (body: collectionId)         -- 서버가 소유자를 조회해 처리
GET  /feed/collections/{collectionId}/shelf -- 같은 선반의 다른 공개 Collection
GET  /follows                              -- 서버가 follow 테이블로 조회
```

- Collection id는 발행 시 공개 대상이므로 노출되어도 정책 위반이 아닙니다. 사용자 열거와는 성격이 다릅니다.
- 향후 Shelf 전용 URL(새로고침·공유 가능한 페이지)이 필요해지면 그때 `member.public_id UUID`를 추가합니다. 기존 행은 일괄 UPDATE로 채울 수 있습니다.

### 5.4 공개 Keyword의 전제

타인에게 Keyword를 제공한다는 것은 Context의 일부가 마스킹된 형태로 공개된다는 의미입니다. 안전성은 다음 전제에 의존합니다.

- Keyword는 `ai.keyword_preset`의 사전 정의 목록에서만 선택됩니다.
- AI가 목록 밖 코드를 반환하면 매핑 단계에서 폐기합니다.

AI가 Keyword를 자유 생성하도록 변경하면 이 전제가 깨지며, 인명 등 식별 정보가 노출될 수 있습니다.

### 5.5 타인 조회 쿼리

```sql
-- Collection 상세: context 본문을 조회하지 않음
SELECT r.id AS record_id, r.created_at,
       p.name, p.address, p.lat, p.lng, p.place_url
FROM collection_record cr
JOIN record r ON r.id = cr.record_id AND r.deleted_at IS NULL
JOIN place  p ON p.id = r.place_id
WHERE cr.collection_id = :collectionId AND cr.deleted_at IS NULL
ORDER BY cr.created_at DESC;

-- Keyword: record_id 목록으로 일괄 조회 후 애플리케이션에서 그룹핑
SELECT ct.record_id, kp.code, kp.label
FROM context ct
JOIN ai.context_keyword ck ON ck.context_id = ct.id
JOIN ai.keyword_preset  kp ON kp.id = ck.keyword_id
WHERE ct.record_id IN (:recordIds)
  AND ct.deleted_at IS NULL
  AND kp.active = true
GROUP BY ct.record_id, kp.code, kp.label;
```

진입 전 권한 검사는 `is_published = true`, `deleted_at IS NULL`, 소유자 미탈퇴입니다. 실패 시 403이 아니라 **404를 반환**합니다. 403은 리소스의 존재를 알려주므로 익명성에 불리합니다.

## 6. 핵심 트랜잭션

### 6.1 Record 생성

```text
Kakao place_id로 Place 조회
→ 없으면 Place 저장
→ Record 생성
→ 첫 Context 생성
→ 커밋
→ (비동기) 임베딩·Keyword 생성 요청
```

- 맥락 없이 장소만 저장할 수 없습니다. 첫 Context는 필수입니다(정책 5.2).
- Place 저장은 동시 요청에 대비해 `INSERT ... ON CONFLICT (kakao_place_id) DO NOTHING` 후 재조회합니다.
- 임베딩과 Keyword는 비동기 생성이므로 Record는 커밋 즉시 활성 상태가 됩니다. 생성 완료 전에는 Keyword가 비어 있을 수 있으며, 화면은 이 상태를 허용해야 합니다.

### 6.2 삭제된 Place의 재저장

```text
Record 생성과 동일
```

- 과거 Record를 재활성화하지 않습니다. 삭제된 행은 그대로 두고 새 Record를 INSERT합니다.
- 활성행 부분 유니크 인덱스가 이를 허용합니다.
- 과거 Context, 임베딩, Collection 연결은 복구하지 않습니다. 사용자 관점에서는 새 Record 생성으로 보입니다.

### 6.3 외부 Place를 내 Record로 추가

```text
kakaoPlaceId로 내 활성 Record 조회
→ 없으면 6.1 Record 생성 (201 RECORD_CREATED)
→ 있으면 기존 Record에 Context 추가 (200 CONTEXT_ADDED)
```

동일 User·Place에 중복 Record를 만들지 않습니다(정책 5.6). 서버가 판단하며 프론트에 거절을 반환하지 않습니다.

### 6.4 Context 수정

Context는 불변이므로 본문을 갱신하는 경로가 없습니다. 수정 요청은 추가와 삭제의 조합으로 처리합니다.

```text
Record 잠금
→ 새 Context 생성          ← 반드시 추가가 먼저
→ 기존 Context 소프트 삭제 + 파생 데이터 파기
→ 커밋
→ (비동기) 새 Context의 임베딩·Keyword 생성
```

- **순서가 중요합니다.** 삭제를 먼저 하면 마지막 Context를 수정할 때 활성 수가 0이 되는 중간 상태가 생깁니다. 추가를 먼저 하면 활성 수가 1 → 2 → 1로 움직여 한 번도 0이 되지 않습니다.
- 두 작업을 한 트랜잭션으로 묶습니다.
- **"마지막 Context는 삭제할 수 없다"는 삭제 유스케이스의 규칙이며, 수정에는 적용하지 않습니다.** 수정은 별도 유스케이스로 구현하고 삭제 API를 재사용하지 않습니다.
- 새 Context는 새 `id`와 새 `created_at`을 가집니다. 사용자 관점에서는 수정이지만 데이터 관점에서는 새 맥락입니다.
- Context 추가·수정 시 `record.updated_at`을 갱신합니다(최신 활동 시각 기록).

### 6.5 Context 삭제

```text
Record 잠금
→ 활성 Context 수 확인
→ 2개 이상: 해당 Context 소프트 삭제 + 파생 데이터 파기
→ 1개    : 삭제하지 않고 409 거절 (DELETE_CONFIRMATION_REQUIRED)
           impact(recordDeleted, collectionIds) 반환
```

- 빈 Record는 허용하지 않으므로, 마지막 Context는 이 API로 삭제할 수 없습니다.
- 프론트엔드는 409의 `impact`를 기반으로 "기록도 함께 사라집니다"(연쇄 Collection이 있으면 그 사실 포함)를 안내하고, 사용자 확인 후 **Record 강제 삭제 API**(`DELETE /records/{recordId}/force`, 6.6)를 호출합니다.
- 이 규칙은 삭제 유스케이스에만 적용됩니다. 수정(6.4)은 추가를 먼저 하므로 해당하지 않습니다.

**Record를 잠그는 이유** — 활성 Context 수를 세고, 그 결과로 분기하고, 쓰기 때문입니다. 잠금이 없으면 두 트랜잭션이 각각 "2개니까 삭제해도 된다"고 판단한 뒤 서로 다른 Context를 지워 활성 수가 0이 될 수 있습니다. 서로 다른 Context를 지우므로 Context 행 잠금은 겹치지 않으며, 두 작업이 공유하는 유일한 대상이 부모인 Record입니다.

### 6.6 Record 삭제 (일반·강제)

프론트는 이 Record가 어떤 Collection의 마지막 Record인지 알 수 없으므로 서버가 DB에서 확인합니다.

**일반 삭제 — `DELETE /records/{recordId}`**

```text
Record 잠금
→ 포함된 활성 Collection 조회 (collection_record.record_id 인덱스)
→ 해당 Record가 마지막인 Collection 식별
→ 있음: 삭제하지 않고 409 거절 (DELETE_CONFIRMATION_REQUIRED, impact.collectionIds 반환)
→ 없음: 아래 삭제 절차 수행
```

**강제 삭제 — `DELETE /records/{recordId}/force`** (409 안내 후 사용자 확인을 받은 프론트가 호출)

```text
Record 잠금
→ Record 소프트 삭제
→ 활성 Context 전체 소프트 삭제
→ CollectionRecord 연결 소프트 삭제
→ 마지막 Record였던 Collection 소프트 삭제
→ 남은 Collection의 record_count 갱신
→ ai.context_embedding / ai.context_keyword 행 삭제
```

- 연쇄 삭제 대상이 없어도 강제 삭제는 정상 수행합니다(일반 삭제와 동일 결과).
- AI 파생 데이터는 즉시 파기합니다. 소유는 AI 파트이나 트랜잭션 일관성을 위해 백엔드가 직접 삭제합니다. 동일 인스턴스이므로 단일 트랜잭션으로 처리됩니다.

### 6.7 Collection에서 Record 제거

```text
Collection 잠금
→ 활성 연결 수 확인
→ 2개 이상: 연결만 소프트 삭제 + record_count 감소
→ 1개    : 삭제하지 않고 409 거절 (DELETE_CONFIRMATION_REQUIRED)
```

- 어느 경우에도 **원본 Record는 유지**합니다. 연결만 끊깁니다.
- 빈 Collection은 허용하지 않으므로, 마지막 연결은 이 API로 제거할 수 없습니다. 프론트엔드는 409를 받아 "컬렉션도 함께 사라집니다"를 안내하고, 확인 후 Collection 삭제 API(`DELETE /collections/{collectionId}`, 6.8)를 호출합니다.
- Collection을 잠그는 이유는 6.5와 같습니다. 활성 연결 수 판단과 `record_count` 갱신이 같은 경합을 갖기 때문입니다.

### 6.8 Collection 삭제

```text
Collection 소프트 삭제
→ CollectionRecord 연결 소프트 삭제
→ 원본 Record 유지
```

### 6.9 회원 탈퇴

```text
member 소프트 삭제
→ social_account 소프트 삭제 + provider_user_id 마스킹
→ Record·Context 소프트 삭제
→ Collection·CollectionRecord 소프트 삭제
→ 해당 User가 생성한 Follow 소프트 삭제
→ 해당 User를 대상으로 하는 Follow 소프트 삭제
→ ai 파생 데이터 파기
→ Feed·Library 노출 즉시 제외
```

- Place는 유지합니다.
- 데이터량이 많으면 이벤트 발행 후 배치로 분리할 수 있습니다. 이 경우에도 Feed·Library 노출 제외는 `member.deleted_at`으로 즉시 반영됩니다.

## 7. 구현 주의점

- **활성 필터 누락 주의.** Collection과 Follow도 소프트 삭제 대상이므로 `deleted_at IS NULL`을 빠뜨리면 삭제된 데이터가 Feed에 노출됩니다.
- **유니크는 반드시 부분 유니크.** 전체 유니크로 정의하면 삭제 후 재저장이 실패합니다.
- **운영자 복구도 새 INSERT로.** 소프트 삭제를 하는 이상 운영자는 사실상 복구 능력을 갖지만, `deleted_at`을 `null`로 되돌리면 부분 유니크 충돌이 발생할 수 있습니다. 복구가 필요하면 데이터를 읽어 새로 INSERT합니다. 정책과 구현이 한 경로만 갖게 됩니다.
- **AI 워커는 쓰기 직전 Context 생존을 확인해야 합니다.** 처리 중에 사용자가 Context를 삭제하면 결과가 고아 데이터로 남습니다. `context.deleted_at IS NULL`을 확인하고 동일 트랜잭션에서 INSERT하거나, 삭제 시 파생 데이터 정리를 한 번 더 수행합니다. 이것이 버전 컬럼을 대신하는 유일한 방어 지점입니다.
- **자연어 검색은 필터 후 벡터 연산.** `context.member_id`로 후보를 좁힌 뒤 벡터 연산을 수행합니다.
- **임베딩 인덱스는 HNSW.** 삭제된 벡터를 남기면 인덱스가 부풀고 recall이 저하됩니다. 즉시 파기 원칙은 정합성뿐 아니라 성능 요건이기도 합니다.
- **임베딩 차원은 스키마에 고정됩니다.** 모델 변경 시 차원이 달라질 수 있으므로 `embedding_profile` 기준으로 재생성 대상을 판별합니다.
- **공용 Place와 카카오 장소 변경.** MVP는 `kakao_place_id`가 변하지 않는다고 가정하고, Place 행을 저장 시점 스냅샷으로 취급합니다. Place는 개인 소유가 아니라 **전체 회원이 공유하는 공용 데이터**이므로 다음을 지켜야 합니다.
  - **기존 Place 행을 프론트 전달값으로 갱신하지 않습니다.** 한 행이 여러 사용자의 Record에 연결되어 있어, 갱신하면 다른 사용자 기록의 장소 정보까지 바뀝니다. 동일 `kakao_place_id` 재전달 시 기존 행을 그대로 재사용합니다.
  - 가정이 깨지는 시나리오와 수용 범위: 폐업 후 재등록·카카오 병합 등으로 id가 바뀌면 동일 실제 장소가 별개 Place로 중복 저장될 수 있고(사용자 관점 영향 미미), 장소로 내 Record를 찾는 조회는 miss 후 신규 저장 경로로 빠지므로 오류가 아닙니다. 상호 변경·이전·폐업은 스냅샷에 반영되지 않으며 최신 정보는 `place_url`로 카카오맵에 위임합니다.
  - **자동 병합 금지.** 이름·좌표 근접 기반 동일성 판정은 확률적이라(한 좌표 다수 점포, 층·호 정보 부재) 잘못 병합 시 타인 장소에 맥락이 붙는 사고가 됩니다. 중복이 실제 문제가 되면 운영자 수동 병합 도구로 대응합니다.
  - 향후 갱신을 도입할 경우 Record에 저장 시점 장소명 스냅샷 컬럼이 필요하며, 좌표가 임계값 이상 이동하면 동일 장소가 아닐 수 있으므로 갱신을 보류하는 가드가 필요합니다.
- **Feed·Library 조회는 다단 삭제 상태 확인.** member, collection, collection_record, record의 삭제 상태를 모두 확인해야 합니다.
- **`.gitattributes`와 무관하게 마이그레이션은 Flyway로.** 스키마 변경 이력을 남기고 JPA `ddl-auto`는 `validate`로 둡니다.
