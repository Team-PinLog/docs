# PinLog 파트 간 요구사항

이 문서는 AI·Backend 파트의 설계가 성립하기 위해 **다른 파트(Front, Infra)가 수행해야 하는 요구사항**을 모읍니다. 각 요구사항의 근거는 공용 계약 [AI 설계](05_AI_설계.md)와 정책 문서에 있으며, 여기서는 파트별로 필요한 항목만 정리합니다.

Front·Infra 레포는 이 저장소에서 직접 관리하지 않으므로, 해당 파트는 이 문서를 요구사항 명세로 참조합니다.

## 1. Front 요구사항

### 1.1 Context 수정 시 식별자 변경 (필수)

Context는 불변 엔티티입니다. 수정은 내부적으로 **기존 Context 삭제 + 새 Context 생성**으로 처리되며, **응답에 새 `contextId`가 반환**됩니다([AI 설계](05_AI_설계.md)의 Context 불변성).

Front는 다음을 지켜야 합니다.

- 수정 응답의 **새 `contextId`를 반영**합니다.
- 구 `contextId`로 재조회하지 않습니다.
- 구 `contextId`를 로컬 캐시 키로 계속 사용하지 않습니다.
- URL이나 선택 상태가 Context id를 포함한다면 새 id로 교체합니다.
- Optimistic Update 시 새 id가 오기 전 구 id를 영구 확정하지 않습니다.

사용자에게는 "수정"으로 보이지만 데이터 관점에서는 새 맥락입니다.

### 1.2 AI 미완료 상태 처리 (필수)

AI 분석은 저장 이후 비동기로 진행되므로, Keyword가 아직 없는 상태가 정상입니다.

- Keyword **빈 배열은 정상 응답**입니다. 로딩 실패나 오류로 처리하지 않습니다.
- Context 생성·수정 직후 Keyword가 잠시 비어 있을 수 있으며, 잠시 후 표시됩니다.
- 수정 재분석 중에는 해당 Context가 자연어 검색·Keyword 응답에서 일시적으로 제외될 수 있습니다.
- **AI 실패가 기본 기능을 막지 않습니다.** Place·Record·Collection은 Keyword 없이도 정상 표시합니다.

### 1.3 Feed 이벤트 전송 (필수)

Feed 추천 결과에 대한 이벤트 중 **`CLICK`·`SAVE`는 Front가 전송**합니다([API 명세](08_API_명세.md)의 `POST /feed/events`).

- payload에 `requestId`와 `position`을 포함합니다.
- `IMPRESSION`은 Feed 응답 생성 시 서버가 직접 기록하므로 Front가 전송하지 않습니다.
- 수집 항목과 저장 구조는 back 파트의 Feed 문서를 따릅니다.

### 1.4 Keyword 공개 등급 표시 (확인 필요)

Keyword는 `PUBLIC` / `PRIVATE_ONLY` / `BLOCKED` 세 등급을 가집니다([익명 SNS 공개정책](04_익명SNS_공개정책.md)).

- 타인 화면에는 `PUBLIC`만 노출됩니다(서버가 필터).
- 본인 화면에는 `PUBLIC`과 `PRIVATE_ONLY`가 제공됩니다.
- **확인 필요**: 본인 화면에서 `PRIVATE_ONLY`를 시각적으로 구분해 표시할지 여부는 제품 결정이 필요합니다.

## 2. Infra 요구사항

### 2.1 pgvector 확장 (필수, 블로킹)

AI 벡터 검색은 pgvector 확장에 의존합니다. 현재 운영 PostgreSQL 이미지(`postgres:16-alpine`)와 로컬 개발 이미지에는 pgvector가 포함되어 있지 않아, 확장 없이는 AI 파트가 로컬·운영 어디서도 동작하지 않습니다.

- 운영·로컬 PostgreSQL 이미지를 pgvector 포함 이미지(예: `pgvector/pgvector:pg16`)로 교체하거나, initdb 단계에서 확장을 설치합니다.
- `CREATE EXTENSION IF NOT EXISTS vector`가 `ai` 스키마 대상 마이그레이션 전에 수행되어야 합니다.

### 2.2 `ai` 전용 DB 계정 분리 (필수)

계약상 **FastAPI(AI 워커)는 `core.*`에 접근하지 않으며 DB 권한이 `ai` 스키마로 한정**됩니다([AI 설계](05_AI_설계.md) 접근 금지 경계). 현재 자격증명이 단일 계정(`postgres-credentials`)이라 이 경계가 문서상 약속일 뿐 강제되지 않습니다.

- `ai` 스키마 전용 DB role을 만들고 `GRANT`로 권한을 `ai` 스키마에 한정합니다.
- FastAPI는 이 전용 계정으로 접속하고, `core.*`에 대한 권한을 부여하지 않습니다.
- 네트워크 정책만으로는 스키마 경계를 강제할 수 없으므로 DB 권한 레벨에서 처리합니다.

### 2.3 LLM·Embedding API Key (필수)

Keyword 판정과 임베딩 생성은 외부 LLM·Embedding API를 호출합니다.

- API Key를 SealedSecret으로 관리합니다.
- FastAPI 배포에 주입합니다.

### 2.4 내부 API 노출 차단 (필수)

`/internal/v1/*`는 Spring–FastAPI 내부 통신 전용이며 Client에 노출되면 안 됩니다([AI 설계](05_AI_설계.md) API 보안).

- FastAPI 서비스는 공용 마이크로서비스 차트를 쓰더라도 **`ingress.enabled: false`**로 등록합니다(차트 기본값이 `true`이므로 명시적으로 꺼야 함).
- 서비스 간 인증을 적용합니다.

### 2.5 Embedding Profile 단일 주입 (필수)

Embedding Profile은 Spring과 FastAPI에 각각 하드코딩하지 않고 **배포 설정의 단일 원본에서 주입**합니다([AI 설계](05_AI_설계.md) 모델과 Profile).

- 공통 ConfigMap 등으로 Profile 문자열을 제공하고, Spring·FastAPI가 같은 키를 읽습니다.

### 2.6 서비스 등록 (필수)

- `apps/prod/*`에 back·ai 서비스를 등록합니다. 현재 등록된 서비스가 없습니다.
- AI 서비스는 위 2.4에 따라 ingress를 비활성화합니다.

| 항목 | 파트 | 성격 |

| 2.2 `ai` DB 계정 | Infra | 경계 강제 |
