# PinLog 정책 정의서

## 1. 기본 원칙

1. 사용자가 장소를 저장한 이유와 개인적 맥락은 비공개 데이터입니다.
2. 다른 사용자에게는 발행된 Collection과 공개 가능한 정보만 제공합니다.
3. 다른 사용자의 실명, 닉네임, 소셜 계정 등 신원 정보는 공개하지 않습니다.
4. Context 없는 Record와 Record 없는 Collection은 허용하지 않습니다.
5. 사용자 소유 데이터는 소프트 삭제하며 사용자 복구 기능은 제공하지 않습니다.
6. Record 수정은 기존 Record를 소프트 삭제하고 새 Record를 생성하는 방식으로 처리합니다.
7. Record 재생성 시 기존 Collection 연결을 새 Record로 승계합니다.
8. AI 처리는 저장이 완료된 뒤 비동기로 수행하며, 저장과 발행을 지연시키지 않습니다.

## 2. 계정 및 인증

- MVP 로그인은 Google, Kakao, Naver 소셜 로그인만 제공합니다.
- 자체 아이디·비밀번호 로그인과 비밀번호 찾기·변경은 제공하지 않습니다.
- 서비스 User와 SocialAccount를 분리합니다.
- 소셜 인증 수단은 `provider + provider_user_id` 조합으로 식별합니다.
- 최초 가입 시 다음 내용을 안내하고 필수 약관 동의를 받습니다.
  - Collection 생성 시 자동 발행
  - Context 원문 비공개
  - Place, Keyword, Record 생성일 등 공개 가능 정보
  - Shelf 단위 익명 Follow 구조

## 3. Place

- 장소 검색과 지도 표시는 카카오맵 API를 사용합니다.
- Place는 실제 지점을 표현하는 공용 데이터이며 특정 User의 소유가 아닙니다.
- 검색 결과 조회만으로 내부 Place를 저장하지 않습니다.
- Record 생성 시 카카오 Place 식별자로 기존 Place를 조회하고, 없을 때 저장합니다.
- Record 삭제와 회원 탈퇴로 Place를 삭제하지 않습니다.

## 4. Record

- Record는 한 User와 한 Place의 연결 단위입니다.
- 한 User는 같은 Place에 활성 Record를 최대 한 개 가질 수 있습니다.
- 활성 Record는 활성 Context를 최소 한 개 가져야 합니다.
- 아직 방문하지 않은 Place도 저장할 수 있으나 저장 이유나 계획을 Context로 작성해야 합니다.
- 소유자는 Place, Context, Keyword, 생성일, Collection 연결을 조회합니다.
- 공개 화면에는 Place, 비식별화 Keyword, Record 생성일만 제공합니다.

### 수정

```text
기존 Record 및 Collection 연결 조회
→ 기존 CollectionRecord 소프트 삭제
→ 기존 Record 소프트 삭제
→ 새 Record 생성
→ 활성 Context 연결
→ Collection 연결 재생성
→ 새 Record 활성화
→ (비동기) 영향받은 Context의 AI 재처리
```

AI 재처리는 새 Record 활성화 이후에 비동기로 진행합니다. AI 처리 완료는 활성화 조건이 아닙니다.

### 삭제

- Record를 제거해도 다른 Record가 남는 Collection은 연결만 제거합니다.
- 마지막 Record인 Collection은 확인 후 Collection도 소프트 삭제합니다.
- 삭제한 Place를 다시 저장하면 과거 연결을 복구하지 않고 새 Record와 Context를 생성합니다.

## 5. Context

- Context는 Record 안의 독립 서술 단위이며 한 Record에 여러 개 존재할 수 있습니다.
- 사용자는 자신의 Context를 생성, 조회, 수정, 삭제할 수 있습니다.
- Context 본문을 수정하면 해당 Context의 AI 분석을 비동기로 다시 수행합니다. 재분석이 끝날 때까지 해당 Context는 자연어 검색과 Keyword 응답에서 일시적으로 제외될 수 있습니다.
- 마지막 Context 삭제 시 Record 삭제 확인을 거칩니다.
- 관련 Record가 Collection의 마지막 Record라면 Collection 삭제 영향도 함께 처리합니다.

## 6. Keyword

- Keyword는 AI가 Place와 Context를 분석하여 서비스 프리셋에서 매핑합니다.
- AI는 프리셋 밖의 Keyword를 자유 생성하지 않습니다. 사람 이름, 구체적 회사·학교, 정확한 개인 일정, 식별 가능한 사건은 공개 프리셋으로 만들지 않습니다.
- 일반적인 관계 표현은 유지할 수 있습니다.
- 사용자는 Keyword를 조회만 하며 직접 생성·수정·삭제할 수 없습니다.
- Keyword는 `PUBLIC`, `PRIVATE_ONLY`, `BLOCKED` 세 등급을 가집니다. 타인에게는 `PUBLIC`만 공개하고, `PRIVATE_ONLY`는 본인에게만 제공하며, `BLOCKED`는 본인에게도 제공하지 않습니다.
- 매칭되는 Keyword가 하나도 없을 수 있으며 이는 오류가 아닙니다.
- AI 분석이 완료되지 않았거나 실패한 경우 Keyword만 생략하고 Place·Record·Collection은 정상 제공합니다.

Keyword 등급별 계산 범위와 처리 방식은 [AI 설계](05_AI_설계.md)를 따릅니다.

## 7. Collection

- Collection은 Record를 1개 이상 묶은 사용자 정의 그룹입니다.
- Place나 Context를 직접 담지 않습니다.
- 동일 Record를 한 Collection에 중복 추가할 수 없습니다.
- 제목은 필수이며 최대 20자입니다.
- 설명 필드는 없습니다.
- 생성 즉시 자동 발행되며 MVP에서 발행 취소와 비공개 전환은 없습니다.
- 발행 중에도 제목과 Record 구성을 수정할 수 있고 변경은 공개 Collection에 즉시 반영됩니다.
- Collection과 내부 Record는 Record `createdAt` 오름차순 정책을 사용합니다.
- 마지막 Record 제거 시 확인 후 Collection을 소프트 삭제합니다.
- Collection 삭제 시 원본 Record는 유지합니다.

## 8. Shelf, Library, Follow

- User마다 Shelf를 하나 가집니다.
- Shelf는 해당 User가 발행한 Collection 목록입니다.
- Library는 내 Shelf와 팔로우한 Shelf를 관리하는 개인 공간이며 공개 프로필이 아닙니다.
- Follow 대상은 User나 Collection이 아니라 Shelf입니다.
- 자신의 Shelf Follow와 동일 Shelf 중복 Follow는 허용하지 않습니다.
- Follow 별칭은 선택값이며 최대 20자입니다.
- 공백 제거 후 빈 값은 `null`로 처리하고 동일 별칭 중복은 허용합니다.
- 팔로워·팔로잉 목록은 제공하지 않으며 수치는 본인만 조회합니다.

## 9. Feed

- Feed의 추천 단위는 발행된 Collection입니다.
- 탈퇴 User와 모든 소프트 삭제 데이터는 노출하지 않습니다.
- 다른 사용자의 Context 원문은 제공하지 않습니다.
- 신고·차단과 개별 Record 추천은 MVP에서 제공하지 않습니다.

## 10. 삭제 및 탈퇴

회원 탈퇴 시 User, SocialAccount, Record, Context, Collection, CollectionRecord, Shelf 및 관련 Follow를 소프트 삭제합니다.

탈퇴한 User의 Shelf와 Collection은 다른 사용자의 Library와 Feed에서 즉시 제외합니다. Place는 유지합니다.
