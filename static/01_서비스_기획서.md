# PinLog 서비스 기획서

## 1. 서비스 개요

| 항목 | 내용 |
|---|---|
| 서비스명 | PinLog / 핀로그 |
| 서비스 형태 | 맥락 기반 장소 기록·검색 및 익명 장소 큐레이션 SNS |
| 팀 규모 | 6인 |
| 개발 기간 | 4~5주 |
| 핵심 타깃 | SNS나 지도에서 장소를 탐색·저장하고 다시 찾는 20~30대 사용자|
| 핵심 기록 단위 | Record |
| 서술 단위 | Context |

PinLog은 사용자가 Place를 Context와 함께 Record로 저장하고, 

이후 자연어로 다시 찾으며, 다른 사용자의 익명 Collection에서 새로운 Place를 발견하는 서비스입니다.

## 2. 문제 정의

장소 정보와 기억은 SNS 저장 목록, 메시지, 블로그, 사진첩 등에 흩어집니다.

- 방문 전에는 왜 저장했는지, 누구와 가려 했는지 다시 찾기 어렵습니다.
- 방문 후에는 장소명보다 메뉴, 분위기, 동행자, 경험만 부분적으로 기억하는 경우가 많습니다.
- 사람 중심 SNS에서는 낯선 사용자의 장소 취향과 테마별 큐레이션을 중심으로 탐색하기 어렵습니다.
- 다른 사람에게서 발견한 장소를 자신의 목적과 맥락을 가진 기록으로 전환하기 어렵습니다.

## 3. 핵심 가치

### 맥락이 남는 장소 기록

사용자는 장소만 저장하지 않고 저장 이유, 목적, 동행자, 분위기, 활동, 방문 경험 등을 Context로 기록합니다.

### 장소명이 없어도 찾는 검색

AI 자연어 검색은 사용자의 Place, Context, Keyword를 검색하여 관련 Record를 반환합니다.

### 익명 취향 발견

Feed는 사람의 신원이 아니라 Collection을 추천합니다. 같은 Shelf의 다른 Collection을 확인하고 Shelf를 팔로우할 수 있습니다.

### 발견을 내 기록으로 전환

다른 사용자의 Collection에서 발견한 Place에 자신의 Context를 작성하여 개인 Record로 저장합니다.

## 4. 핵심 사용 모드

| 사용 모드 | 사용자 목표 | 대표 질의·행동 |
|---|---|---|
| 미래 계획 | 가고 싶은 장소를 당시 계획으로 다시 찾기 | “부산에서 친구들과 가려고 했던 곳” |
| 과거 기록 | 다녀온 장소를 경험 맥락으로 회상하기 | “부모님과 갔던 조용한 한식당” |
| 장소 발견 | 익명 큐레이션에서 장소를 찾고 내 기록으로 만들기 | Feed 탐색 → Place 선택 → Context 작성 |

세 모드는 별도 데이터 모델이 아니라 동일한 Record–Context 구조를 사용합니다.

## 5. 핵심 사용자

### 미래 계획형

SNS와 메시지에서 장소를 많이 저장하지만 실제 약속 때 저장 이유와 위치를 찾기 어려운 사용자입니다.

### 과거 기록형

긴 리뷰는 부담스럽지만 좋았던 장소를 경험과 함께 남기고 다시 찾고 싶은 사용자입니다.

### 발견 탐색형

장소 취향이 맞는 익명 사용자의 Collection과 Shelf를 탐색하고 싶은 사용자입니다.

## 6. 제품 원칙

- 개인 Context는 기록과 검색에 사용되지만 다른 사용자에게 공개하지 않습니다.
- 공개는 Collection과 Shelf를 중심으로 구성합니다.
- 발견한 Place는 타인의 기록을 복사하지 않고 사용자의 새 Context로 개인화합니다.
- 기능 포함·제외 기준은 `04_MVP_기능범위.md`를 따릅니다.
